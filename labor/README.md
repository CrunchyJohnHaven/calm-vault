# labor-orchestrator-v2

The system John uses to cast offshore-labor nets across 5-10 channels in parallel, measure throughput per `(channel × task_type)`, and scale the winners.

> **Thesis (John, 2026-05-11):** *"With offshore labor we can do 10x more I bet. Maybe 100x. We must cast a wide net and test carefully for throughput and maximize."*

This repo subdirectory is a **state machine + a paste-buffer**. It does *not* talk to any platform's API, it does not auto-respond to applicants, and it does not hold payment credentials. John copy-pastes. The orchestrator just tracks, classifies, and recommends.

---

## 5-minute quickstart

```bash
cd labor
pip install -r requirements.txt    # PyYAML only

# 1. Cast a 4-channel net for QA work, $25 per channel, $100 total
./orc post --task qa_read_through \
           --channels upwork,fiverr,reddit_forhire,onlinejobs_ph \
           --budget 100 \
           --test-id test_001_qa_4channel
# -> prints 4 paste-ready posting bodies + logs 4 rows to state/postings.jsonl

# 2. As applicants reply, classify them
./orc inbox
# -> reads ~/AllData/inbox/listener.jsonl, classifies, appends to applications.jsonl

# 3. Award one contract per channel (John hires manually). Then, on delivery:
./orc score worker_upwork_anya \
            --test test_001_qa_4channel --channel upwork \
            --task qa_read_through \
            --quality 8.5 --hours 2.5 --paid 25.00 \
            --accepted-at  2026-05-12T01:10:00Z \
            --delivered-at 2026-05-13T14:00:00Z \
            --notes "8/8 salted bugs, 0 false positives, 3 non-salted defects"

# 4. After all four deliver, look at the leaderboard
./orc leaderboard --write     # writes labor/leaderboard.md
./orc recommend --task qa_read_through
./orc dashboard --write       # writes labor/DASHBOARD.md
```

State is plain JSONL because John wants to grep it.

## The CLI: `orc`

| command | what it does |
|---|---|
| `orc post --task T --channels A,B,C --budget N` | Prints paste-ready postings for each channel; logs to `postings.jsonl`. Use `--per-channel` to make budget per-channel instead of total. |
| `orc inbox [--source PATH] [--dry-run]` | Reads `~/AllData/inbox/listener.jsonl` (or `--source`), classifies each row by channel & task_type, scores applicants `shortlist|reject|maybe`, appends to `applications.jsonl`. |
| `orc score WORKER --task T --channel C --quality 0..10 --hours H --paid $` | Records a completed work item to `work.jsonl`. Prefer `--test TEST_ID` to link back to the originating posting. |
| `orc leaderboard [--write]` | Prints the channel × task_type table to stdout; with `--write`, also overwrites `labor/leaderboard.md`. |
| `orc recommend --task T` | Prints the current best channel for that task type, with sample size and `$/quality-point` rationale. |
| `orc dashboard [--write]` | Renders the single-page dashboard with active postings, open applications, in-progress work, pending payments, burn, and the single "what to do next" line. |

Every command is read-only on `task_types.yaml` and `channels.yaml`. Only `post`, `inbox`, and `score` write state.

## What this is NOT

- **Not an API integration.** Upwork/Fiverr/etc. all forbid automated posting. John posts manually.
- **Not a payment system.** Payment happens out-of-band (Wise, PayPal, Upwork escrow, etc.). The `score` command and `payments.jsonl` only *record* what John already paid.
- **Not an auto-responder.** Applications are classified, never replied to automatically. John approves every hire.
- **Not a database.** State is plain JSONL on disk. John can grep / wc / cat / diff at will.

## Quality metric — why `$/quality-point`?

The leaderboard's headline metric is `mean_cost_per_hr ÷ mean_quality_0_to_10`. Lower is better. This trades off two things we genuinely care about — cost and quality — into one number we can rank channels by.

Channels with `n<3` are reported honestly (the leaderboard prints `n` next to every cell) but are not eligible for 10x scale-up — that requires `n≥3` deliveries with `quality ≥ 7.0`. See `_next_step` in `orc` for the exact gating logic, and `tests/test_001_qa_4channel.md` for the "decision rule for winner."

## Extending

- **Add a task type** → append to `task_types.yaml` with the schema documented there.
- **Add a channel** → append to `channels.yaml` with the schema documented there.
- **Add a posting template** → drop `postings/<task>/<channel>.md`. `orc post` finds it automatically. Channel keys in filenames must match `channels.yaml` keys.

There is intentionally no autogenerator for postings. The whole point is that the language differs by channel (Reddit's `[Hiring]` tag, Fiverr's seller-finds-buyer flow, Upwork's fixed-vs-hourly trade-off). Hand-write each one once.
