# Everest 67 — Disclosure Response Schema

*Phase VI — Disclosure Semantics. Prereq: Everest 66, 45.*

## Overview

A disclosure response is the operator's atomic answer to a counterparty's (C's) disclosure request. It encodes the outcome of evaluating one or more predicates against the principal's commitments, bundled with cryptographic proofs, temporal anchors, and operator authorization. The response schema is designed for **verifiability**, **privacy**, and **auditability**: the counterparty can cryptographically confirm that the operator honestly evaluated the requested predicates and bound the result to a public, immutable timeline.

The response has two outcomes:
1. **proof_provided**: one or more predicates evaluated; committed bits, ZK proofs, and anchors included.
2. **no_proof**: consent, predicate definition, or other preconditions failed; body uniformly empty (per Everest 77).

## Canonical JSON Schema

```json
{
  "schema_version": "1.0.0",
  "request_id": "<UUIDv4 echoing the E66 request_id>",
  "response_id": "<UUIDv4 unique per response>",

  "result": {
    "outcome": "proof_provided" | "no_proof",
    "predicates": [
      {
        "predicate_id": "<canonical predicate_id>",
        "predicate_parameters": { },
        "disclosed_bit": "True" | "False" | "Indeterminate",
        "freshness_seconds": "<integer>",
        "commitment": "<Pedersen commitment, compressed Ristretto255 bytes (hex)>",
        "proof": "<Bulletproofs/Schnorr proof bundle (hex)>"
      }
    ],
    "combinator_result": {
      "combinator": "AND" | "OR" | "SINGLE" | "SELECTIVE",
      "final_bit": "True" | "False" | "Indeterminate",
      "final_freshness": "<integer>"
    }
  },
  "chain_head": "<32-byte sha256 hex>",
  "anchor_proofs": {
    "sigsum": "<Sigsum inclusion proof + signed tree head (hex)>",
    "roughtime": "<Roughtime signed timestamp from N servers (hex)>"
  },
  "operator_vc": "<operator's CredexAI VC pointer or inline VC>",
  "counterparty_vc_fingerprint": "<SHA-256 of C's VC for binding>",
  "request_nonce": "<echo of C's request nonce>",
  "operator_signature": "<Ed25519 signature over canonical response body>"
}
```

## Field Semantics

### Top-Level Envelope

- **schema_version**: Identifies the response format. Incremented only on breaking changes.
- **request_id**: UUID copied from the disclosure request (E66). Ties the response to a specific request; enables request–response correlation without revealing the request body to third parties.
- **response_id**: Unique UUID generated by the operator for this response. Enables auditing and prevents response replay when combined with response_ts and operator_signature.
- **response_ts**: ISO8601 timestamp, cryptographically anchored via Roughtime and Sigsum (see **anchor_proofs**). Establishes a reliable wall-clock reference for predicate freshness and immutable ordering.

### Result Block

**outcome**: Either "proof_provided" or "no_proof". Controls the shape and contents of the response.

**predicates** (proof_provided only):
- Array of evaluated predicates. Each entry represents one predicate from the request, evaluated against the principal's current commitments and templates.

Per-predicate fields:
- **predicate_id**: Canonical identifier from E26 (e.g., "zk_distance_le_100km", "template_id_match"). Enables the verifier to look up the ZK circuit (E65) and interpretation rules.
- **predicate_parameters**: JSON object echoing the parameters from the request (e.g., `{"threshold_km": 100, "template_id": "..."`). Ensures transparency; allows verifier to confirm evaluation was performed with intended parameters.
- **disclosed_bit**: The outcome of the predicate evaluation: "True", "False", or "Indeterminate" (if data is insufficient or consent was partial).
- **freshness_seconds**: Integer representing how old the underlying commitment is (in seconds from response_ts). Verifier uses this to assess whether the commitment is stale per C's freshness requirement (E66).
- **commitment**: Pedersen commitment value (compressed Ristretto255, hex-encoded). This is the point on the elliptic curve that binds the predicate state. Verifier confirms it is on-curve and verifies the zero-knowledge proof against it.
- **proof**: Bulletproofs + Schnorr composition proof bundle (hex-encoded). Circuit is determined by predicate_id (E65). Proof is non-interactive and can be verified offline.

**combinator_result**:
- **combinator**: Logical combinator from the request ("SINGLE", "AND", "OR", "SELECTIVE"). If the request evaluated a single predicate, combinator is "SINGLE".
- **final_bit**: The combined outcome ("True", "False", or "Indeterminate").
- **final_freshness**: The freshness of the combined result, typically the minimum of the component predicates.

### Temporal Anchors

**chain_head**: A 32-byte SHA-256 hash of the latest block (or log entry) in the operator's internal append-only ledger. This hash is submitted to both Sigsum and Roughtime for cryptographic anchoring (see E93, E94). The verifier can audit the operator's historical ledger by fetching the signed tree head from Sigsum and confirming chain_head is included.

**anchor_proofs**:
- **sigsum**: Inclusion proof that chain_head is in the Sigsum transparency log. Includes the signed tree head from Sigsum operators, proving the hash was publicly logged at a specific tree size and verified by the Sigsum ecosystem (E93).
- **roughtime**: Roughtime signed timestamp (aggregated over N Roughtime servers) for response_ts. Prevents backdating; proves response_ts is genuinely recent and not forged (E94).

### Authorization & Binding

**operator_vc**: Credential issued by CredexAI (or a trusted issuer per E68). Contains the operator's public key and role/tier claims. Can be a full VC inline or a pointer (e.g., URL or DID) for retrieval by the verifier.

**counterparty_vc_fingerprint**: SHA-256 hash of the counterparty C's credential. Used to bind the response to a specific counterparty identity, preventing unauthorized sharing or impersonation (per E68 operator identity binding). Verifier checks this fingerprint against their own VC to confirm the response was issued for them.

**request_nonce**: Echo of the nonce C sent in the disclosure request (E66). Verifier computes the SHA-256 hash of this nonce and confirms it was included in C's request_id or sent as an opaque value; ensures response is not a replay or a response to a different request.

**operator_signature**: Ed25519 signature over the **canonical response body** (see Canonicalization below). Verifier retrieves operator's public key from operator_vc, verifies the signature, and confirms the response was not tampered with.

## No-Proof Case (E77 Integration)

When outcome is "no_proof":
- **result.predicates**: Empty array.
- **result.combinator_result**: Omitted or null.
- **chain_head, anchor_proofs**: Omitted or minimal (operator_signature still required for integrity).
- HTTP status code: **204 No Content** (per E77).
- Counterparty receives no information about *why* the response was denied (consent missing, predicate undefined, rate-limited, etc.). This **uniform silence** prevents inference attacks and protects the principal's privacy.

The operator_signature on the (request_id, response_id, response_ts, "no_proof") body proves the operator intentionally refused, and is not a connection failure or timeout.

## Canonicalization for Signing

Before generating operator_signature, the response body is canonicalized using **RFC 8785 JSON Canonical Serialization (JCS)**:
- Keys sorted lexicographically.
- Whitespace removed.
- Numeric values as integers or floats (per RFC 8785 rules).
- No trailing newlines.

The signature is computed over the UTF-8 bytes of the canonical form. Verifier must re-canonicalize the received JSON and verify the signature; any deviation (whitespace, key order, number representation) will cause signature verification to fail.

## Verification Flow (Counterparty)

1. **Parse JSON**: Validate against the schema. Reject if required fields are missing or types are incorrect.
2. **Verify VC**: Fetch or parse operator_vc. Confirm:
   - VC is signed by a trusted issuer (CredexAI).
   - VC is not expired (check iat, exp claims).
   - VC public key is valid (Ed25519 or similar).
3. **Verify operator_signature**: Canonicalize the response body (excluding operator_signature). Use operator's public key from VC to verify the Ed25519 signature. Reject if signature fails.
4. **Verify request_nonce**: Confirm the nonce matches the one C sent in the request. Prevents response hijacking.
5. **Verify chain_head anchor (Sigsum)**: Fetch the signed tree head from a known Sigsum operator (E93). Confirm that chain_head is present in the log at the claimed tree size. Verifies the operator is not equivocating or revising history.
6. **Verify response_ts anchor (Roughtime)**: Validate the Roughtime quorum signature for response_ts (E94). Requires aggregation of signatures from N independent Roughtime servers (typically N=3–4). Confirms response_ts is genuine and not backdated.
7. **Verify counterparty_vc_fingerprint**: Compute SHA-256 of C's own VC. Confirm it matches the value in the response. Ensures response is intended for C.
8. **For each predicate (if outcome = "proof_provided")**:
   a. **Verify commitment on-curve**: Confirm the commitment is a valid Ristretto255 point.
   b. **Verify ZK proof**: Use the predicate_id to look up the ZK circuit (E65). Execute the proof verifier with the commitment, proof, and predicate_parameters. Reject if proof fails.
   c. **Check freshness**: Compare freshness_seconds to C's freshness requirement (from the request). Reject if stale.
9. **Verify combinator_result** (if outcome = "proof_provided" and combinator != "SINGLE"):
   - Recompute the logical combination of individual predicate bits using the specified combinator (AND, OR, SELECTIVE).
   - Confirm the result matches final_bit.
   - Verify final_freshness is consistent with component freshness values.
10. **Attestation**: If all checks pass, C can now trust the disclosed bits and assert them in downstream decisions (e.g., access control, transaction approval).

## Privacy Properties

- **Verifier learns**: The outcome of each requested predicate (True, False, Indeterminate) and the freshness window.
- **Verifier does NOT learn**:
  - Raw biometric data (e.g., raw fingerprint, iris scan).
  - Distance values or other continuous metrics (only "True" or "False" of the thresholded predicate).
  - Principal's vocabulary or speech patterns (if applied to NLP predicates).
  - Internal chain contents or ledger structure.
  - Why a "no_proof" response was issued (uniform silence per E77).
- **Unlinkability**: Each predicate in the response uses a fresh Pedersen randomness, so commitments for the same predicate across different requests are unlinkable. An observer cannot correlate requests.
- **Operator cannot equivocate**: The operator's signature on the response and the Sigsum/Roughtime anchors create an immutable record. The operator cannot later deny issuing the response or change the outcome without detection.

## Summary

Everest 67 defines the atomic unit of disclosure: a cryptographically bound, temporally anchored response that proves the operator honestly evaluated requested predicates without revealing sensitive underlying data. The schema enforces verifiability (through operator signatures and public logs), privacy (through ZK proofs and uniform silence on refusal), and auditability (through temporal anchors and immutable ledger records). Together with E66 (request) and E77 (refusal semantics), it completes the disclosure conversation.
