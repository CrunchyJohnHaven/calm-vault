# Everest 64 — Predicate Test Corpus

*Phase V — Predicate Authoring. Prereq: Everest 63.*

## Overview

Everest 64 establishes the test corpus for all predicates in the ZKBB protocol. This corpus provides the ground truth for predicate determinism validation (E63) and proof circuit correctness (E65), ensuring that every predicate behaves consistently across all input variations and edge cases. The corpus is structured, peer-reviewed, and executable in CI pipelines.

## Corpus Architecture

### Directory Structure

```
test_corpus/
  <predicate_id>/
    <test_case_id>.json          # Individual test cases
    <test_case_id>.json
    ...
  adversarial/
    <predicate_id>/
      <adversarial_id>.json      # Fuzzing and red-team inputs
```

### Test Case Schema

Each test case is a JSON file conforming to this structure:

```json
{
  "case_id": "string",
  "predicate_id": "string",
  "description": "string describing what this case tests",
  "input": {
    "chain_snapshot": { },
    "parameters": { },
    "now": "ISO8601 timestamp"
  },
  "expected_output": {
    "bit": "True | False | Indeterminate",
    "freshness": "seconds since snapshot captured"
  },
  "rationale": "string explaining why this input should yield this output",
  "peer_reviewers": ["reviewer_1@...", "reviewer_2@..."],
  "status": "active | deprecated",
  "notes": "optional field for disputed or edge cases"
}
```

## Corpus Composition Requirements

### Minimum Volume

- Minimum 30 hand-crafted test cases per predicate
- Cases distributed across four categories:
  * Positive cases (10): inputs where the predicate SHOULD return True
  * Negative cases (10): inputs where the predicate SHOULD return False
  * Indeterminate cases (5): inputs where tri-valued logic requires Indeterminate
  * Edge cases (5): boundary conditions, malformed inputs, temporal edges, clock skew

### Predicate-Specific Coverage

**in_baseline_24h (E55):**
- Baseline match scenarios (principal's latest biometric report within vocab)
- Vocab mismatch (principal's report uses obsolete vocabulary)
- Restedness failures (baseline fails due to insufficient sleep)
- Health issues preventing baseline validation
- 24-hour window boundaries (exact cutoff times)
- Missing or incomplete reports

**biometric_match_within (E56):**
- Cases at threshold τ=0.1, 0.3, 0.5, 0.7
- Distances strictly above threshold (should match)
- Distances strictly below threshold (should not match)
- Distances exactly at threshold (requires explicit semantics)
- Distance calculations with different metric spaces

**bank_teller_note_active (E58):**
- Codeword present and valid
- Codeword absent
- Codeword expired (freshness exceeded)
- Codeword revoked
- Wrong codeword presented
- Multiple codewords in same note (precedence rules)

**cognitively_atypical_baseline (E59):**
- Declaration present and in-scope
- Declaration absent
- Declaration superseded by newer declaration
- Declaration expired
- Multiple conflicting declarations

**mental_state_unusual (E60):**
- Biometric drift detected (high distance from baseline)
- Vocabulary mismatch (principal's report uses different vocab)
- Health issue flag raised
- Explicit flag in declaration
- Combinations of the above

**principal_consents_to_disclose (E57):**
- Consent grant (active, matches counterparty class)
- Consent expired (timestamp exceeded)
- Consent revoked (explicit revocation in chain)
- Class match (counterparty in permitted class)
- Class mismatch (counterparty outside permitted class)
- Per-counterparty override (explicit exception in chain)

### Synthetic vs. Realistic Balance

- 60% synthetic cases: designed to precisely hit specific edge conditions
- 40% realistic cases: derived from anonymized actual chains, all principals pre-consented for use as test material
- Each realistic case includes source reference (anonymization hash) for audit trails

## Chain Snapshot Generation

Each test case includes a deterministic chain snapshot (a frozen state of the principal's data at a specific `now` timestamp). Snapshots are generated by:

1. **Template-based generation:** Scripts in `test_corpus_generators/` build snapshots from templates specifying chain history, biometric reports, declarations, etc.
2. **Deterministic time:** Every snapshot uses a fixed `now` timestamp; no live clock
3. **Reproducibility:** Generator scripts are versioned and checked into the registry repo
4. **Immutability:** Snapshots are static JSON; they do not mutate during test execution

Generator scripts support parameterized templates for:
- Biometric report sequences (varying distance, vocab, freshness)
- Declaration timelines (expiry, revocation, supersession)
- Chain history depth and structure
- Time zone and clock-skew injection

## Peer Review Process

Each test case is reviewed by at least two outside reviewers (distinct from the case author):

### Review Checklist

1. **Does the case test what its rationale claims?** Verify that the input is designed to isolate the behavior described in the rationale.
2. **Is the expected output correct?** Verify that the expected output semantically aligns with the predicate definition (E51, E52, etc.) and the input.
3. **Is the case independent?** Verify that the case does not implicitly rely on the behavior of other predicates or external systems.
4. **Are chain snapshots well-formed?** Verify that snapshots conform to the ZKBB protocol's schema.
5. **Is the rationale clear?** Verify that a third party can understand why this case matters.

### Dispute Resolution

- Reviewers who disagree on expected output enter discussion
- Discussion is tracked in the case's `notes` field
- Consensus is required for activation; consensus is recorded in `peer_reviewers` list
- Until consensus is reached, the case is marked `status=controversial` and excluded from CI
- Controversial cases may be split (two distinct cases with different expected outputs) or discarded if unresolvable

## Test Execution

### Determinism Harness Integration (E63)

The determinism harness executes every predicate × every active test case in every CI run:

1. **Deterministic execution:** Predicate code is isolated; no side effects, no external state
2. **Pass/fail reporting:** Results visible in CI artifacts (pass if actual output matches expected output)
3. **Coverage metrics:** CI reports corpus coverage (% of cases passing, % with disagreement)
4. **Regression detection:** Changes to predicate code are immediately validated against the full corpus

### Execution Artifact Structure

```
ci_artifacts/
  predicate_<id>/
    <test_case_id>/
      actual_output.json
      pass_fail.txt
      stderr.txt
    summary.json
```

## Corpus Maintenance

### Adding Cases

1. Author creates new test case JSON in `test_corpus/<predicate_id>/`
2. Case includes at least two proposed peer reviewers
3. Case is opened as a PR to the registry repo
4. Reviewers verify case per the review checklist
5. After consensus, case is merged with `status=active`
6. CI immediately executes the new case

### Deprecation, Not Deletion

- Cases are never deleted from the corpus
- To retire a case, set `status=deprecated` with justification in `notes`
- Deprecated cases are excluded from CI but preserved in git history
- History of deprecated cases aids in understanding predicate evolution

### Versioning

- Corpus is versioned by commit hash in the registry repo
- Each E63 CI run records the corpus commit used
- Proof circuit (E65) is tied to a specific corpus version

## Adversarial Corpus

### Purpose

Adversarial cases are inputs designed to find bugs, not validate correctness. They supplement the main corpus.

### Structure

```
test_corpus/<predicate_id>/adversarial/
  <fuzzing_id>.json
  <red_team_id>.json
```

### Generation Methods

- **Fuzzing (E85):** Automated property-based tests that generate malformed inputs, boundary values, and stress-test conditions
- **Red-team review (E87):** Security-focused review teams deliberately attempt to construct inputs that violate the predicate's invariants

### Handling

- Adversarial cases do NOT have expected outputs; they are passed to predicate code and any exception is logged
- Exceptions are reviewed to determine if they indicate a bug (expected) or a predicate defect (unexpected)
- Cases that reveal bugs are converted to regular test cases with appropriate expected outputs and added to the main corpus

## Success Criteria

Everest 64 is complete when:

1. All six predicates (E55–E60) have ≥30 active test cases each
2. Each case has ≥2 peer reviewers and active status
3. Coverage distribution meets the requirements (positive, negative, indeterminate, edge)
4. All cases execute deterministically in E63 harness
5. Realistic cases are derived from principal-consented anonymized chains
6. Adversarial corpus exists and surfaces zero unexpected exceptions on initial red-team run
