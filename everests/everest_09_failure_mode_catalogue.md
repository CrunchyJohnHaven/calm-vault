## Chain Failure Modes

**F01 — Stale Chain (no recent records).** The operator has not received a fresh self-report or biometric sample from the principal for a period exceeding the predicate's freshness window. A verifier checking `chain_head.timestamp` against the current time will find a gap and reject the proof as insufficiently recent. The operator detects this before proof generation via checks in Everest 55/56 predicates, which enforce maximum-age thresholds. If a principal tries to disclose with a stale chain, the verifier's freshness-window check (anchored to Roughtime consensus time) will fail and the proof is rejected. The response is straightforward: abort the proof and ask the principal to provide a fresh sample.

**F02 — Chain Head Not Anchored to Sigsum Within Freshness Window.** A chain head published to Sigsum is only tamperproof if the inclusion proof and signed timestamp are fresh. If the operator publishes to Sigsum but the inclusion proof is lost or corrupted, or if the timestamp falls outside the verifier's accepted freshness window, the proof lacks the transparency-log anchor that binds it. The verifier queries the Sigsum inclusion endpoint; if no proof is returned or the timestamp is stale, the chain head is rejected. The response is to abort the disclosure and retry anchoring in a subsequent session once the chain head has been re-published.

**F03 — Roughtime Quorum Failure / Clock Skew.** Roughtime is used to verify that the chain-head timestamp is not tampered with by comparing it against at least N independent time servers. If fewer than N servers respond (quorum lost) or if the reported timestamps differ by more than a predefined threshold (clock skew), the Roughtime aggregator in the operator's vault signals an error before proof generation. The operator detects this and rejects the disclosure request, asking for fresh Roughtime consensus. This failure is local and does not leak data — it is purely a freshness guardrail.

**F04 — Sigsum Operator Collusion (All Witnesses Lie).** Sigsum's security model depends on N-of-M transparency-log witnesses being honest; if all collude, they can forge an inclusion proof. The verifier's defense is to independently query multiple Sigsum operator endpoints; if ≥3 witnesses report conflicting inclusion proofs or timestamps, the verifier detects the conflict. In this case, the protocol escalates to governance (Everest 93): operators and the principal community agree that the set of Sigsum witnesses is compromised and migrate all Calm Witness chain heads to a new transparency log. This is a high-severity failure (S2) because detection is non-trivial and requires out-of-band governance, but the protocol does not accept a false-positive disclosure — the verifier will reject.

**F05 — Local Chain File Corruption (Partial Overwrite).** If the principal's device suffers a partial write failure, a record in `user_state.jsonl` might be corrupted. The CLI tool `calm-witness verify-chain` (Everest 28) walks the chain from genesis, computing `sha256(prev_hash + payload)` for each record and checking it against the stored `record_hash`. A single bit flip in the corrupted record will cause the hash to mismatch. The operator detects this and alerts the principal. Recovery attempts are made using encrypted replicas (Everest 32–33); if a replica is intact, the chain is reconstructed from it and Sigsum's inclusion proofs.

**F06 — Hash Chain Break (prev_hash Mismatch).** If a record's `prev_hash` field does not match the prior record's `record_hash`, the chain is broken. This might result from an attempted retroactive edit or from corruption. The `verify-chain` tool detects it immediately and halts all disclosures. The principal must reconcile with the Sigsum history to determine which record is authentic (the one Sigsum witnessed) and which is spurious. Once reconciled, the chain is rebuilt from the Sigsum-attested heads and replicas.

**F07 — Replay of Old Chain Head as if Fresh.** An attacker captures a valid chain head H from a past session and tries to reuse it in a new disclosure. To prevent this, the verifier binds a nonce from the counterparty's current disclosure request into the proof (Everest 70). The commitment includes this nonce; if the attacker replays the proof with a different nonce, the commitment is invalid. The verifier detects the mismatch and rejects. This is S2 (high) because replay attacks are feasible for an attacker with network access, but the protocol detects and stops them.

## Consent Failure Modes

**F08 — Expired Consent Record.** Consent records include an `expiry` timestamp; a consent can be time-bounded (e.g., "valid for 30 days"). The predicate evaluator `p3: principal_consents_to_disclose(p, c)` checks `consent.expiry < now()`. If the check fails, the predicate evaluates to false and no proof is generated. The operator detects this before attempting to construct the proof and informs the principal that consent has expired. The principal can then re-authorize if desired.

**F09 — Revoked Consent In-Flight (Revocation Between Proof Generation and Verification).** A principal revokes consent, creating a revocation record in the vault. The operator generates a proof based on the not-yet-revoked consent. In transit, the verifier checks a CRL or OCSP-equivalent endpoint (maintained in Everest 75) and learns that the consent has been revoked. The verifier rejects the proof. This is detected at the counterparty side, and the disclosure log (Everest 72) records the revocation-induced failure for the principal's audit.

**F10 — Consent Record Signed by Stale Principal Key.** Consent records are signed by the principal's identity key (part of the CredexAI VC). If that key has since been rotated or revoked, the consent signature is no longer valid. The predicate evaluator checks the signature against the current principal key; if it is stale, the signature fails verification and the consent is invalid. The response is to ask the principal to re-authorize using their current key.

**F11 — Counterparty-Class Mismatch (Consent for Class X, Request from Class Y).** The principal may consent to disclosure to the "journalistic" counterparty class but not the "governmental" class. The counterparty C provides its CredexAI VC, which carries a class label. The predicate `p3` checks whether the class label matches the consent record's class; if not, the predicate is false. The proof is not generated. The response is to abort and allow the principal to decide whether to grant an exception.

## Biometric / Template Failure Modes

**F12 — Biometric Drift Beyond Tolerance (Legitimate Drift, Not Attack).** Handwriting and speech naturally evolve over weeks and months due to aging, illness, accent drift, or changes in motor habits. The drift-modeling mechanism (Everest 39) uses exponential-moving-average to update the template on confirmed-baseline sessions. If drift accumulates faster than the EMA can accommodate, the distance of a new sample to the old template exceeds the per-principal threshold τ. The predicate `p2: biometric_match_within(τ)` evaluates to false. The operator detects this and alerts the principal; a re-enrollment is triggered (Everest 18). Until re-enrollment, the principal cannot generate proofs. This is S3 because the protocol correctly rejects the proof and no false disclosure occurs; it is a degradation of UX.

**F13 — Sample Replay (Same Sample Used in Two Disclosures).** An attacker with access to the vault tries to reuse a past biometric sample (e.g., the same handwriting strokes or voice transcript) in a new session. The sample uniqueness check (Everest 50) commits a hash of each sample to the chain. If the same sample is submitted in a second session, its commitment matches a prior chain record and the chain rejects it. The operator detects this and alerts both the principal and the security team. This is S2 because the attack is caught, but the attempt itself indicates a vault breach or operator compromise.

**F14 — Template Overdue for Re-enrollment.** Templates have an expiry policy (e.g., valid for 90 days). The operator tracks `template.created_at` and refuses to generate proofs after the template expires unless a new one is enrolled. The principal is alerted in advance; if they do not re-enroll by the deadline, the operator rejects all disclosure requests post-deadline. This forces freshness and prevents reliance on very-old biometric baselines.

**F15 — Substitution Attack (Different Human at Capture Time).** An attacker replaces the principal during a biometric sample and captures their own handwriting or voice. The biometric distance from the attacker's sample to the enrolled template (from the true principal) is very large — far beyond any legitimate drift. The distance proof (Everest 45) commits to a distance that is too large and fails the range check. The verifier rejects the proof. This is S1 because it is a fundamental identity-binding failure; the protocol detects it and prevents false disclosure.

**F16 — Liveness Detector Spoofed (Pre-recorded Strokes or Transcript).** An attacker replays a pre-recorded handwriting sequence (e.g., from a prior session) or a pasted transcript. The liveness detector (Everest 49) examines hardware-level entropy: for strokes, micro-variations in pen-tip pressure (which vary with each real stroke); for transcripts, inter-keystroke timing entropy (which differs each time a human types). Pre-recorded samples lack this variation. The operator's capture pipeline rejects them as non-liveness. The principal and operator are alerted. This is S1 because liveness is essential to binding the proof to the principal's current state.

**F17 — Cross-Template Inconsistency Proof Failure.** During re-enrollment, the old template v_n and new template v_n+1 should be consistent — the new one should be a drift of the old, not a replacement by a different human. The cross-template consistency proof (Everest 48) verifies this. If the proof fails (e.g., templates are too dissimilar), re-enrollment is aborted. The principal is forced to repeat the enrollment ceremony; an investigation is launched into why the templates diverged.

## Key / Identity Failure Modes

**F18 — Operator Key Exfiltration.** The Calm operator signs every disclosure proof using their CredexAI-issued identity key. If this key is exfiltrated (e.g., via a supply-chain compromise), an attacker can forge proofs. The detection mechanism is revocation: CredexAI can revoke the operator's VC. Any verifier that checks the VC (Everest 68) will see it is revoked and reject all proofs signed by that operator. Once revocation is detected, the principal rotates to a different operator. All outstanding proofs from the compromised operator are considered untrusted. This is S1 because an attacker can forge proofs, but detection and response are swift.

**F19 — Principal Master Key Loss (Recovery Path).** The principal loses access to the vault (device destroyed, forgotten password, etc.). If the principal is alive and was enrolled with witnesses (Everest 20), those witnesses can sign attestations of the principal's identity. Combined with Sigsum's immutable record of the chain heads and replicas, the vault can be reconstructed. The recovery protocol (Everest 23) is followed. This is S3 because recovery is possible and documented.

**F20 — CredexAI VC Expired or Revoked.** Both the operator and principal have CredexAI VCs that are time-limited and can be revoked. If a VC expires or is revoked before re-issuance, the corresponding proofs are rejected by verifiers. The principal or operator must refresh or re-authenticate with CredexAI to obtain a new VC. No disclosure can proceed until this is resolved.

**F21 — Subverted Enrollment Device at Template-Creation Time.** If the hardware or software used to capture the initial biometric templates is compromised, the templates themselves are poisoned. An attacker can craft samples that match the fake templates but are from a different human. This is explicitly out-of-scope for the protocol (Everest 1 threat model) — it is a physical-security problem. The only defense is the Everest 11 enrollment ceremony, which requires an airgap, witness presence, and other procedural safeguards. If this is discovered post-facto, all proofs using that template are considered compromised and are audited for any prior disclosures.

## Protocol Failure Modes

**F22 — Replay of a Valid Calm Witness Proof.** An attacker captures a valid proof and tries to use it in a new disclosure request to the same or a different counterparty. The proof includes a commitment to a nonce from the counterparty's original request (Everest 70). If the attacker tries to present the proof with a fresh nonce, the commitment does not match and the verifier rejects it. If the attacker presents the proof with the original nonce, the counterparty's system detects the repeat and rejects it on policy grounds. This is S2 because replay is a serious attack, but the protocol detects and prevents it.

**F23 — Coerced Principal (Rubber-Hose; Explicitly Out-of-Scope).** A principal under direct physical or legal coercion is asked to disclose state. This is a universal attack against which no cryptographic protocol can defend. However, Calm Witness includes a mitigating feature: the bank-teller-note codeword (Everest 58). The principal can embed a secret duress phrase in a self-report, which flips a dedicated bit without revealing the phrase to anyone (including the operator). A counterparty trained to recognize duress-bit patterns can escalate without the principal having to explicitly ask for help. This is S1 in severity but explicitly acknowledged as out-of-scope for cryptographic defense.

**F24 — Coerced Operator.** An operator under coercion is pressured to issue a false proof (e.g., claiming the principal is in baseline when they are not). However, the proof still binds to the vault state: to forge a false proof, the operator would have to retroactively edit the vault's self-report records and re-publish the chain head to Sigsum. This is not possible without subverting Sigsum itself. Any attacker observing the vault (via Everest 72 audit logs and Sigsum history) will see the inconsistency. The operator cannot unilaterally issue a false proof; the best an attacker can do is force the operator to sign a true proof based on corrupted vault state. The principal's audit log reveals the disclosure and can be used for legal recourse.

**F25 — Network Partition During Disclosure.** Proof generation and verification are largely local and offline. A network partition only affects the operator's ability to publish the chain head to Sigsum. If the partition is temporary, the operator retries once connectivity is restored. If the partition is prolonged, the operator can generate a proof using a locally cached Sigsum anchor from the last successful publication; the proof's freshness window is determined by the timestamp of the last successful anchor. This is S3 — it degrades UX but does not create a security failure.

**F26 — Verifier Downgrade Attack (Counterparty Refuses to Verify).** The counterparty agent receives a valid Calm Witness proof but chooses to ignore it and instead request raw biometric data or other evidence from the principal. This is not a failure of the Calm Witness protocol — it is a failure of the counterparty's policy. Calm Witness is designed to be used by counterparties that trust the protocol; counterparties that do not can always fall back to their existing (less privacy-preserving) methods. The governance (Everest 80) and the principal's choice of counterparties are the levers to prevent this.

**F27 — Side-Channel Leak (Timing, Power, Fault Injection at Proof Generation).** The Pedersen commitment and Σ-protocol operations are vulnerable to side-channel attacks if implemented naively. The distance comparison and commitment opening can leak information via timing variations or power consumption. The defense is to use constant-time cryptographic libraries (e.g., libsodium) and to zeroize all intermediate values after use. The Rust production implementation (Everest 81) is audited (Everest 90) for timing safety and is compiled with constant-time flags.

**F28 — Compromised Counterparty Re-Discloses Bit Publicly.** The counterparty receives a valid proof and learns the disclosed bit (e.g., "principal is in baseline"). They then publish this bit publicly, causing reputational harm to the principal. Calm Witness has no cryptographic defense against this — the bit is intended for the counterparty's use and the protocol trusts them not to abuse it. The only defense is preventive: the principal only discloses to counterparties they trust (Everest 7, 73). If a breach occurs, the principal's audit log (Everest 72) shows exactly which counterparty received the disclosure, providing a basis for legal recourse.

## Policy Failure Modes

**F29 — Predicate Evaluation Non-Determinism (Same Input → Different Bit).** A predicate is evaluated twice over the same vault state and yields different bits, violating the protocol's core assumption that predicates are deterministic. This might result from a bug in the predicate code, floating-point rounding errors, or non-deterministic hashing. The CI harness (Everest 63) runs every predicate against a frozen test corpus; any variance is immediately flagged. If a non-determinism is discovered in production, the buggy predicate version is deprecated (Everest 53 predicate registry), all proofs issued with that version are tagged, and the principal and affected counterparties are notified. No false disclosure is guaranteed because the verifier can independently re-evaluate the predicate.

**F30 — Stealth Disclosure (Everest 78) Misfires (False-Positive Duress Signal).** The duress codeword is a principal-private phrase embedded in a self-report. A false-positive occurs if the principal's normal baseline affective vocabulary happens to contain a phrase that resembles the codeword. For example, if the codeword is "Charlie is here" and the principal normally writes "Charlie just arrived," the detector might trigger falsely. The mitigation is to choose a rare, principal-specific codeword (e.g., a nonsense phrase or a reference only the principal knows). If false-positives become frequent, the principal can disable stealth disclosure and use only explicit (pull-model) disclosures. This is S1 in severity because a false-positive duress signal could cause the counterparty to take inappropriate action, but it is jointly mitigated by the principal's codeword choice and the option to disable.

## Severity Ranking Philosophy

Failures are ranked S1 (catastrophic), S2 (high), S3 (medium), or S4 (low). We prioritize **false-positive disclosure of principal harm** (a false-positive safety signal that exposes the principal to unintended action or reputational damage) as S1 — above all else. False negatives (the protocol failing to disclose when it should) are also S1 because they can hide duress or state changes material to the counterparty's decision. Attacks that the protocol detects and prevents cleanly are S3 at worst, because no actual harm accrues. Failures that degrade UX without creating a security breach are S4. We reason that it is better to abort a disclosure and disappoint a principal than to issue a proof that leaks biometrics or misrepresents the principal's state. The bank-teller-note primitive is only trustworthy if it reliably attests to the principal's own sense of their state — never to a counterparty's interpretation of them.
