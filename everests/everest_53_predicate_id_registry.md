# Everest 53 — Predicate ID Registry

*Phase V — Predicate Authoring. Prereq: Everest 52.*

## Overview

The Predicate ID Registry is a public, append-only Git repository that serves as the single authoritative source of truth for all Calm Witness predicates. It maps each `predicate_id` to its canonical JSON specification, human-readable documentation, reference implementation, zero-knowledge circuit definition, test corpus, provenance metadata, and lifecycle status. The registry enforces anti-fragmentation, enables offline operation, and provides clear versioning semantics for predicate evolution.

The canonical registry lives at `github.com/CrunchyJohnHaven/calm-witness-predicates` under Apache-2.0 licensing. All Calm Witness operators, provers, and verifiers derive their predicate truth from this single repository.

## Registry Structure

The repository is organized by predicate name, with each predicate's evolution tracked through semantic versioning:

```
predicates/
  in_baseline_24h/
    v1.0.0/
      spec.json           # canonical JSON spec
      spec.md             # human-readable doc
      reference.rs        # canonical Rust evaluator
      circuit.txt         # ZK circuit definition
      test_corpus/
        golden_001.json   # golden input/output pairs
        golden_002.json
        ...
      provenance.json     # who proposed, who reviewed, when
      status.json         # active | deprecated | superseded
    v1.0.1/
      spec.json
      spec.md
      reference.rs
      circuit.txt
      test_corpus/
        ...
      provenance.json
      status.json
  biometric_match_within/
    v1.0.0/
      ...
    v2.0.0/
      ...
  principal_consents_to_disclose/
    v1.0.0/
      ...
  bank_teller_note_active/
    v1.0.0/
      ...
  cognitively_atypical_baseline/
    v1.0.0/
      ...
  mental_state_unusual/
    v1.0.0/
      ...
```

Each predicate_id is content-addressed according to Everest 52 canonical form semantics: the ID is derived deterministically from the spec.json contents, guaranteeing that identical specs receive identical IDs and any spec mutation invalidates the predicate_id.

## Entry Composition

Every predicate version directory contains:

**spec.json**: The canonical, minimal JSON representation of the predicate. This is the normative specification—all other formats (spec.md, reference.rs, circuit.txt) are derived from or consistent with this artifact. The spec includes input schema, output schema, parameter constraints, evaluation semantics, and references to precursor or successor predicates.

**spec.md**: Human-readable markdown documentation of the predicate's purpose, inputs, outputs, semantics, and use cases. Intended for auditors, operators, and downstream system designers who need to understand what the predicate asserts without reading JSON or Rust.

**reference.rs**: A canonical Rust implementation of the predicate evaluator. This serves as the ground truth for predicate behavior and must be proven equivalent to the ZK circuit. The reference implementation includes comprehensive logging and error handling to support debugging and audit trails.

**circuit.txt**: The zero-knowledge circuit definition in a human-inspectable format (e.g., Circom or Halo2 circuit syntax). This circuit is proven equivalent to the Rust reference via Everest 63 (determinism harness). Proofs generated by this circuit carry the predicate_id as a commitment to this exact circuit definition.

**test_corpus/**: Golden input-output pairs that serve as regression tests and as examples for operators and implementers. Each golden test includes a human-readable label (golden_001.json, etc.) and must pass both the reference.rs evaluator and the circuit definition. Test corpus is versioned with the predicate; breaking changes to tests require a new version.

**provenance.json**: Metadata recording who proposed the predicate, who reviewed it, timestamp of adoption, and a link to the governance PR where the predicate was approved. This creates an auditable chain of custody and enables later attribution and governance analysis.

**status.json**: A lightweight document recording the predicate's lifecycle state and any successor or predecessor references. States are:
- `active`: The predicate is current and accepts new proofs.
- `deprecated`: The predicate is stable but no longer used for new consents; existing proofs remain valid for verification.
- `superseded`: The predicate has been replaced; this entry names the new predicate_id and includes a migration guide if applicable.

## Predicate Lifecycle and Versioning

Each predicate evolves through semantic versioning (major.minor.patch). Within a predicate name (e.g., in_baseline_24h), multiple versions may coexist in the registry:

- **v1.0.0** is the initial published version.
- **v1.0.1** patches a bug in the test corpus or reference implementation without changing observable semantics.
- **v1.1.0** adds an optional input parameter or refines output interpretation in a backward-compatible way.
- **v2.0.0** changes the core semantics, input schema, or output format—this is a new predicate_id.

Version numbers are independent per predicate. The registry enforces that within a predicate, only the latest minor series (e.g., 1.x) is marked `active`; earlier minors transition to `deprecated` when a new minor is published. Major version bumps to v2+ create a new predicate_id and set the old predicate to `superseded`.

When a new version is adopted, the registry commit includes a pointer from the new version to its predecessor, enabling downstream tools to understand migration paths and historical relationships.

## Anti-Fragmentation Guarantee

The Calm Witness protocol depends on all parties trusting the same predicate definitions. The registry enforces this through a single canonical source of truth:

**Single repository**: There is one authoritative registry at the canonical URL. Operators pull predicates from this repository.

**Forks are allowed but discouraged**: Apache-2.0 licensing permits forking, but any fork is immediately detectable as non-canonical. Counterparties verifying proofs must explicitly specify which registry they trust; the canonical registry is the default. If a fork diverges, downstream verifiers will detect predicate_id mismatches and reject proofs that claim to use predicates from an out-of-band registry.

**Content addressing**: Each predicate_id is a cryptographic hash of spec.json (Everest 52). This means two predicates with the same human name but different specs will have different predicate_ids. There is no ambiguity.

**Sigsum anchoring**: Each commit to the registry is published to Sigsum, a public append-only log. This enables anyone to detect a registry fork: the Sigsum log provides a monotonic timeline of all canonical commits, so any privately held "version" of the registry that diverges from Sigsum is provably forked.

## Governance Process

New predicates are added through a structured review process (composed with Everest 54, the audit process, not yet bagged):

1. **Proposal**: A contributor opens a PR with a complete predicate definition including spec.json, spec.md, reference.rs, circuit.txt, test_corpus, and provenance.json.

2. **Review**: At least two independent external reviewers examine the proposal. Reviewers check that the spec is clear, the reference implementation matches the circuit, the test corpus is comprehensive, and the predicate serves a genuine use case.

3. **Adoption**: A maintainer of the registry merges the PR after review passes. The merge commit triggers automated signing and Sigsum publication.

4. **Maintenance**: The registry maintainer role is multi-signature: Calm, Koushik, and one independent third party must all sign off on any commit. This prevents single-point-of-failure control and embeds trust distribution into the repository's operational structure.

Deprecation and supersession are also governed through pull requests: a maintainer updates status.json to mark a predicate `deprecated` or `superseded`, and the same review and signing process applies.

## Operator Integration

Calm Witness operators integrate the registry in two modes:

**Online mode**: Operators fetch the latest predicate definitions from the canonical registry on startup or on-demand via `calm-witness predicates pull`. The local cache is checked against the registry's Sigsum chain to verify no divergence has occurred.

**Offline mode**: Operators ship with a frozen snapshot of the registry—a cryptographic tarball of all predicates as of a specific git commit. When offline, the operator uses only this snapshot. When coming back online, `calm-witness predicates pull` fetches new predicates and updates the snapshot, verifying the update against Sigsum.

This dual mode enables air-gapped deployments while preventing predicate drift.

## Verifier Integration

Verifiers use the registry to validate proofs. The `calm-witness verify` command implements the following flow:

1. Extract the `predicate_id` from the proof.
2. Look up the predicate_id in the local registry snapshot.
3. If the predicate_id is missing or the version doesn't match the proof's version claim, verification fails with a clear error message indicating which predicate is missing or mismatched.
4. Load the predicate's circuit definition and execute the zero-knowledge proof verification against that circuit.
5. Cross-check the predicate_id in the proof against the content hash of the circuit and spec, ensuring the prover used the canonical definition.

This process ensures that no two parties can disagree about what a predicate asserts.

## Public APIs

The registry is accessible through the following `calm-witness` subcommands:

- **`calm-witness predicates list`**: Display all predicates in the local registry snapshot, with names, versions, and statuses.

- **`calm-witness predicates pull`**: Update the local registry snapshot from the canonical remote. Verifies the update against Sigsum and Sigstore signatures.

- **`calm-witness predicates show <predicate_id>`**: Display the full specification, reference implementation, and test corpus for a given predicate.

- **`calm-witness predicates verify-registry`**: Verify the integrity of the local registry snapshot by checking Sigsum anchors and Sigstore signatures on all release tags.

## Cryptographic Anchoring

**Sigsum publication**: Each registry commit is published to the Sigsum transparency log. The commit hash and metadata are recorded immutably, creating a global timeline of all canonical updates. This prevents a private registry fork from being undetectable.

**Sigstore signing**: Each release tag in the registry is signed using Sigstore. Downstream verifiers can verify the signature to confirm that a predicate was published by the canonical registry maintainers, without needing to trust a centralized CA.

**Content addressing**: Predicate_ids are content-addressed hashes of spec.json (Everest 52), so any modification to the spec changes the ID. Proofs commit to their predicate_id, so changing a predicate after a proof is generated invalidates the proof.
