# Everest 70 — Replay Defense

*Phase VI — Disclosure Semantics. Prereq: Everest 66, 67.*

## Overview

A valid Calm Witness proof is a cryptographic attestation of a principal's consent to disclosure at a specific point in the blockchain and chain state. Once generated, that proof is a bearer token: any party in possession of it can submit it to a verifier and learn the disclosed fact.

The Replay Defense Everest eliminates this vulnerability by binding each proof to the requesting counterparty's nonce, rendering the proof invalid for any subsequent request, any alternative counterparty, and any malicious replay attempt.

## The Threat

An attacker (counterparty, eavesdropper, or network observer) captures a valid proof at time *t₀* and re-submits it at time *t₁* > *t₀*, when one or more of the following is true:

- The principal has revoked consent or changed their disclosure preferences.
- The blockchain has advanced; the chain head is stale relative to *t₁*.
- The disclosed bit has changed (e.g., a score or balance has been updated).
- The counterparty has lost authorization to see the fact (e.g., a lending agreement has expired).

In each case, the proof should no longer be valid. Without replay defense, the attacker succeeds in extracting stale or unauthorized information.

Severity: High. A single captured proof can be weaponized indefinitely.

## The Defense

Bind the counterparty's per-request nonce into the proof's Fiat-Shamir challenge derivation. The proof becomes valid *only* for that specific (counterparty, nonce) pair. Any replay—whether by the same counterparty with a different nonce, or by a different counterparty with any nonce—fails verification.

## Mechanism

### Request-side: Nonce Binding

The counterparty initiates a disclosure request (Everest 66). The request includes:

- **nonce_C**: A 32-byte value, cryptographically fresh per request. Generated by the counterparty's verifier; unique within a reasonable time window (e.g., per-second granularity or higher).
- **request_signature**: The counterparty signs the full request (including nonce_C) with their private key, proving they authored this specific request.

The operator receives the request, validates the signature, and extracts nonce_C.

### Proof Generation: Nonce in Fiat-Shamir Transcript

The operator constructs a Fiat-Shamir transcript for the zero-knowledge proof (Everest 67) as follows:

```
transcript = SHA-256(
  "calm-witness-disclosure-v1" ||
  request_canonical_bytes ||
  chain_head ||
  commitments_vector ||
  nonce_C ||
  ...other_proof_params
)
```

The critical addition is **nonce_C** (the counterparty's nonce) as an explicit field in the transcript. The Fiat-Shamir challenge *e* is derived directly from this transcript:

```
e = SHA-256(transcript) [mod 2^256 or the curve's scalar field]
```

Since nonce_C is bound into *e*, any change to nonce_C produces a different challenge. The proof's response components (r, s) are computed using this challenge *e*; they are mathematically invalid for any other challenge value.

### Verification: Nonce Reconstruction

When a verifier (the counterparty's own verifier, or any third-party auditor) receives the proof, they:

1. Extract the proof components (C_d, range_proof, chain_head, anchor_proof, operator_VC, response_nonce, operator_sig).
2. Reconstruct the Fiat-Shamir transcript **using the nonce that is bound to this specific verification context** (their original nonce_C from the request).
3. Recompute the challenge: e' = SHA-256(transcript').
4. Verify the operator's signature and the zero-knowledge proof against e'.

If the proof was generated for a different nonce_C', the recomputed challenge e' will not match the challenge *e* that was used to generate the proof's responses. Verification fails.

## Operator-side Nonce Tracking

To prevent the counterparty from re-submitting an old proof using the same nonce (e.g., if they retain a cached proof), the operator maintains a **nonce-history log**:

- **Scope**: Sliding window of recent nonces (e.g., 24 hours).
- **On request receipt**: Operator checks if nonce_C has been seen before within the window.
  - If yes: reject the request with a "nonce replay detected" error. The counterparty must generate a fresh nonce.
  - If no: process the request, generate the proof, and record nonce_C in the history with a timestamp.
- **History cleanup**: Entries older than the window are discarded.

This operator-side check is stateless from the counterparty's perspective (the counterparty learns only "nonce in use") but prevents the counterparty from being their own attacker.

## Composition with Other Everests

### Everest 68: Operator VC Signature

The operator's verifiable credential (Everest 68) also commits to the request nonce:

```
operator_VC_claim = {
  disclosure_commitment: C_d,
  request_nonce: nonce_C,
  chain_head: head_hash,
  timestamp: t_operator,
  ...
}
operator_sig = Sign(operator_private_key, operator_VC_claim)
```

The operator's signature is valid only if the VC claim's nonce_C matches the request. A malicious operator cannot forge a different nonce into the claim; the counterparty will detect the mismatch.

### Everest 69: Counterparty VC

The counterparty's own VC binds them to their request signature. The counterparty cannot claim "I didn't generate that nonce" because their VC ties them to their request-signing key. Nonce-binding is thus bound to a known, identified counterparty.

### Everest 72: Logged Disclosure

When the disclosure is logged (Everest 72), the log entry includes request_nonce:

```
log_entry = {
  principal_id,
  counterparty_id,
  request_nonce,
  disclosed_bit,
  timestamp,
  chain_head,
  operator_signature,
  ...
}
```

This creates an immutable audit trail. If the same nonce appears in multiple log entries, an analyst can detect that the operator violated the nonce-deduplication rule and flag the breach.

### Everest 75: Revocation

When the principal revokes consent (Everest 75), the revocation timestamp *t_revoke* is recorded. A proof generated at *t_proof* < *t_revoke* is invalid after revocation, regardless of nonce status. Nonce-binding works in conjunction with revocation: a stale proof is blocked both by revocation logic (time-based) and by nonce logic (if the counterparty generates a fresh nonce after revocation, the operator rejects it outright).

### Everest 76: Rate Limits

Rate-limiting rules (Everest 76) can incorporate nonce-history:

- Per-counterparty nonce rate: e.g., "max N fresh nonces per minute."
- Per-principal disclosure rate: e.g., "max M disclosures of this bit per hour."

Nonce-history feeds into rate-limit buckets, allowing fine-grained control.

### Everest 78: Stealth Disclosure

For push-based disclosures (Everest 78), no counterparty request is present. Instead:

- The operator pre-shares a nonce-seed with the recipient (e.g., derived from a shared secret or via a preamble message).
- For each stealth push, the operator derives a per-recipient nonce using the seed: nonce_push = KDF(seed, push_index, timestamp).
- The recipient can verify the derived nonce and reconstruct the Fiat-Shamir challenge independently.
- Nonce deduplication ensures each push_index is used only once; re-using the same index produces an identical nonce, which the operator rejects on the next request.

## Edge Cases

### Eavesdropper on Counterparty's Network

**Scenario**: An attacker observes the counterparty's request-response exchange on the network, captures the proof, and later attempts to use it.

**Defense**: The proof is bound to the counterparty's nonce_C. To re-submit the proof, the eavesdropper must:

1. Forge a new request with a different nonce_C'.
2. The proof (bound to nonce_C) will not verify against the new request (with nonce_C').

If the eavesdropper attempts to re-use the same nonce_C, the operator's nonce-history check rejects it. The eavesdropper cannot obtain the counterparty's private key to re-sign a request, so all paths to replay are blocked.

### Malicious Operator

**Scenario**: The operator is subverted and attempts to bind an arbitrary nonce into the proof, or to ignore a counterparty's request nonce.

**Defense**: The counterparty's request signature certifies that nonce_C was part of the original request. The operator cannot substitute a different nonce into the Fiat-Shamir transcript without invalidating the request signature. The verifier checks that the operator's claim (in the operator_VC, Everest 68) matches the nonce in the counterparty's original request. A mismatch is evidence of operator misconduct and is logged.

### Operator Nonce-History Compromise

**Scenario**: An attacker gains temporary access to the operator's nonce-history store and deletes or modifies entries.

**Defense**: The nonce-history is a write-once log per entry (or append-only). If an entry is missing or modified, the audit trail (Everest 72) will show a contradiction: the same nonce appears in the disclosure log but not in the operator's nonce-history. This is detected during post-hoc audits and triggers revocation of the operator's VC. Real-time defense: the operator can use a cryptographic accumulator or a tree-based structure to commit to the entire nonce-history, making tampering detectable.

### Counterparty Reusing Nonce Intentionally

**Scenario**: A counterparty generates a proof for nonce_C, then deliberately re-submits a request with the same nonce to obtain an updated proof.

**Defense**: The operator's nonce-history prevents this. The second request with nonce_C is rejected. The counterparty must generate a fresh nonce. This is by design: each proof is tied to a specific request context. If the counterparty needs an updated proof (e.g., because the chain has advanced), they must make a new request with a new nonce.

## Summary

Everest 70 implements replay defense by cryptographically binding each Calm Witness proof to the requesting counterparty's nonce. The nonce is embedded in the Fiat-Shamir challenge derivation, ensuring the proof is valid only for that specific (counterparty, nonce) pair. Operator-side nonce deduplication prevents the counterparty from replaying their own old proofs. Composition with the operator's VC and logging creates an auditable, tamper-evident system.

The defense is forward-compatible with revocation, rate limiting, and stealth disclosure modes. Eavesdroppers and malicious operators are both defeated by nonce binding and request-signature verification.
